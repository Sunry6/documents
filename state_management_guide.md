**状态管理使用指南**

状态管理可以说是 React 开发中最容易拉开代码质量差距的一个领域，重要程度很高。
原因在于它的影响是"乘数效应"——状态设计得好不好，会直接决定组件拆分是否合理、渲染性能是否正常、业务逻辑是否容易追踪。一个状态设计混乱的组件，往往同时伴随着难以维护的 JSX、莫名其妙的 bug 和难以复现的渲染问题。

面向新同事的项目规范 · 适用工具：useState / Context / Jotai · useReducer 作为特定场景补充

| <p>**⚡ TL;DR** 两个核心问题：① 这个状态需要跨组件共享吗？② 状态逻辑需要单独测试吗？</p><p>默认策略：局部状态 → useState（或 object useState）；共享状态 → Jotai atom；不变配置 → Context。</p><p>useReducer 是可选的：只在「需要把状态逻辑提取为纯函数单独测试」时才考虑，其他场景不必用。</p> |
| :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

# **1. 整体思路：先问「共享范围」，再问「复杂度」**

选择状态管理方案时，按这两个问题依次判断：

| <p>第一步：这个状态需要在多个组件之间共享吗？</p><p>│</p><p>├─ 否（只在当前组件用）</p><p>│ │</p><p>│ ├─ 字段简单、独立？</p><p>│ │ → useState</p><p>│ │</p><p>│ └─ 多个字段需要一起更新？</p><p>│ → object useState（首选）</p><p>│ 或 useReducer（当逻辑需要单独测试时）</p><p>│</p><p>└─ 是（需要共享）</p><p>`      `│</p><p>`      `├─ 数据几乎不变化？（主题、语言、初始配置）</p><p>`      `│ → Context + useState</p><p>`      `│</p><p>`      `└─ 动态数据，多个组件读写？</p><p>`            `→ Jotai atom</p> |
| :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

这个顺序很重要：不要一开始就想「我要用 Jotai 还是 useReducer」，而是先确定「这个状态需不需要离开当前组件」。一旦不需要共享，就尽量保持局部；一旦需要共享，就直接上 Jotai，不要绕道 Context 传 props。

# **2. useState — 局部简单状态的首选**

## **2.1 什么时候用**

| **✅ 适合** 状态只在当前组件内用，字段之间相互独立，更新逻辑简单。 |
| :----------------------------------------------------------------- |

- UI 开关状态：modal 的显示/隐藏、dropdown 展开收起、tooltip hover
- 独立的输入值：搜索框文字、数字输入
- 单个 loading 或 error 标志（只在本组件展示时）

示例：

| <p>const [isOpen, setIsOpen] = useState(false)</p><p>const [searchText, setSearchText] = useState('')</p><p>const [count, setCount] = useState(0)</p> |
| :---------------------------------------------------------------------------------------------------------------------------------------------------- |

## **2.2 多字段时：优先考虑 object useState**

当组件内有几个相关联的字段时，不需要直接跳到 useReducer，先试试把它们合并成一个 object：

| <p>// ❌ 多个独立 useState，更新时手动同步容易出错</p><p>const [isSubmitting, setIsSubmitting] = useState(false)</p><p>const [error, setError] = useState(null)</p><p>const [data, setData] = useState(null)</p><p></p><p>// ✅ 合并成 object useState，一次 setState 更新多个字段</p><p>const [fetchState, setFetchState] = useState({</p><p>`  `isSubmitting: false,</p><p>`  `error: null,</p><p>`  `data: null,</p><p>})</p><p></p><p>// 更新：spread 保留其他字段</p><p>setFetchState(prev => ({ ...prev, isSubmitting: true, error: null }))</p> |
| :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

object useState 写法直观，不需要额外 API。如果字段很多、更新逻辑极其复杂，或者你需要单独测试这段逻辑，再考虑 useReducer（见第 4 节）。

# **3. Jotai — 共享状态的主力**

## **3.1 为什么这个项目以 Jotai 为主**

我们项目中同时有 Context 和 Jotai，但两者定位不同。Jotai 解决的核心问题是：多个组件需要订阅同一份会动态变化的数据。

Jotai 相比其他方案的优势：

- **细粒度订阅：**只有用到某个 atom 的组件才会在该 atom 变化时重渲染，不会像 Context 那样全量重渲染
- **原子化：**每个 atom 独立，不需要一个庞大的全局 store，方便按需拆分和复用
- **可在组件外使用：**通过 store.get/set 可以在 API 拦截器、工具函数等组件外读写状态
- **派生状态：**computed atom 让你声明式地描述「这个值怎么从其他 atom 算出来」，不需要在每个组件里重复计算

## **3.2 什么时候用**

| **✅ 适合** 状态需要被多个组件（尤其是不在同一棵组件树上的组件）读取或修改。 |
| :--------------------------------------------------------------------------- |

- 全局用户信息（登录后多处展示用户名、头像、权限）
- 跨页面持久的业务状态（购物车、当前筛选条件、多步骤表单的中间状态）
- 需要在 API 拦截器或工具函数中访问的状态（如 token、locale）
- 用来替代频繁更新的 Context，避免全量重渲染

## **3.3 基础用法**

定义 atom（统一放在 src/store/ 目录）：

| <p>// src/store/userAtom.ts</p><p>import { atom } from 'jotai'</p><p>import type { User } from '@/types'</p><p></p><p>// 基础 atom</p><p>export const userAtom = atom<User | null>(null)</p><p></p><p>// 派生 atom：从其他 atom 计算，只读，自动同步</p><p>export const isLoggedInAtom = atom((get) => get(userAtom) !== null)</p><p>export const isAdminAtom = atom((get) => get(userAtom)?.role === 'admin')</p> |
| :------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

在组件中使用（根据需要选择合适的 hook）：

| <p>import { useAtom, useAtomValue, useSetAtom } from 'jotai'</p><p></p><p>// 只读取 → useAtomValue（只订阅，不返回 setter）</p><p>const user = useAtomValue(userAtom)</p><p></p><p>// 只写入 → useSetAtom（不订阅该 atom，不会因为它变化而重渲染）</p><p>const setUser = useSetAtom(userAtom)</p><p></p><p>// 读写都需要 → useAtom</p><p>const [user, setUser] = useAtom(userAtom)</p> |
| :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| **⚡** 性能提示：能用 useAtomValue 就不用 useAtom；能用 useSetAtom 就不用 useAtom。减少不必要的订阅，组件渲染次数会更少。 |
| :------------------------------------------------------------------------------------------------------------------------ |

## **3.4 object atom：替代 useReducer 的简洁写法**

对于有多个字段的共享状态，用一个 object atom 加 spread 更新，比引入 useReducer 简洁得多：

| <p>// src/store/filterAtom.ts</p><p>interface FilterState {</p><p>`  `keyword: string</p><p>`  `status: 'all' | 'active' | 'done'</p><p>`  `page: number</p><p>}</p><p></p><p>export const filterAtom = atom<FilterState>({</p><p>`  `keyword: '',</p><p>`  `status: 'all',</p><p>`  `page: 1,</p><p>})</p><p></p><p>// 组件中更新（只改其中几个字段）</p><p>const setFilter = useSetAtom(filterAtom)</p><p>setFilter(prev => ({ ...prev, keyword: 'react', page: 1 }))</p> |
| :------------------------------------------------------------------------------------------------------------ | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |

# **4. useReducer — 特定场景下的可选工具**

| <p>**📌 定位** useReducer 不是「状态变复杂就必须用」的工具。在这个项目里，大多数「复杂状态」可以用 object useState 或 Jotai object atom 解决。</p><p>useReducer 真正的优势只有一个：把状态更新逻辑提取为纯函数，方便写单元测试。如果你没有这个需求，大可不用它。</p> |
| :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

## **4.1 什么时候考虑 useReducer**

- 状态逻辑复杂，需要单独提取为纯函数并写单元测试。
- **状态转换有明确的「动作」语义**（如多步骤 wizard、有 undo/redo 的编辑器），用 action 描述比直接 setState 更清晰。
- **当 action type 列表可以充当文档**，让新同事一眼看出「这个状态允许哪些转换」。

| **⚠️ 注意** 如果你发现自己只是想把几个 setState 合并，或者觉得「这个状态字段有点多」——这不是用 useReducer 的理由。object useState 或 object atom 更简单，优先选它们。 |
| :-------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

## **4.2 示例：适合 useReducer 的场景**

多步骤表单（wizard），状态转换逻辑复杂且需要测试：

| <p>type WizardState = {</p><p>`  `step: 1 | 2   | 3</p><p>`  `formData: Record<string, string></p><p>`  `isSubmitting: boolean</p><p>`  `error: string | null</p><p>}</p><p></p><p>type WizardAction =</p><p>`  ` | { type: 'NEXT_STEP' }</p><p>`  ` | { type: 'PREV_STEP' }</p><p>`  ` | { type: 'SET_FIELD'; field: string; value: string }</p><p>`  ` | { type: 'SUBMIT_START' }</p><p>`  ` | { type: 'SUBMIT_ERROR'; error: string }</p><p></p><p>// 纯函数 → 容易单测，不需要 render</p><p>function wizardReducer(state: WizardState, action: WizardAction): WizardState {</p><p>`  `switch (action.type) {</p><p>`    `case 'NEXT_STEP':</p><p>`      `return { ...state, step: Math.min(state.step + 1, 3) as 1 | 2   | 3 }</p><p>`    `case 'SUBMIT_START':</p><p>`      `return { ...state, isSubmitting: true, error: null }</p><p>`    `// ...</p><p>`  `}</p><p>}</p><p></p><p>// 单测：直接调用纯函数，不需要 mount 组件</p><p>// expect(wizardReducer(state, { type: 'NEXT_STEP' })).toEqual({ ...state, step: 2 })</p> |
| :---------------------------------------- | --- | ---------------------------------------------------------------------------------------------------- | -------------------------------------------------------- | -------------------------------- | -------------------------------- | -------------------------------------------------------------- | ----------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |

## **4.3 useReducer vs object useState — 怎么选**

|                        | **object useState**    | **useReducer**                 |
| :--------------------- | :--------------------- | :----------------------------- |
| **写法复杂度**         | 简单，直接 spread      | 需要定义 action 类型和 reducer |
| **适合字段数量**       | 少到中等               | 多，且有复杂转换逻辑           |
| **状态逻辑单测**       | 不方便（绑定在组件里） | 方便（纯函数，直接 import 测） |
| **动作语义明确性**     | 一般                   | 高（每个 action 即文档）       |
| **推荐程度（本项目）** | 优先选择               | 按需使用                       |

# **5. Context — 稳定配置的通道**

Context 在这个项目里定位很窄：传递几乎不会变化的全局配置，主要目的是避免 prop drilling，而不是响应式状态管理。

| ✅ 适合 Context：主题（light/dark）、国际化语言、feature flags（启动时读取一次） |
| :------------------------------------------------------------------------------- |

| <p>⚠️ 不适合 Context：用户输入、实时数据、频繁更新的业务状态。</p><p>原因：Context value 每次变化会让所有消费者重新渲染，即使它们并不关心变化的那部分数据。这类场景请用 Jotai。</p> |
| :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

# **6. 快速参考表**

遇到具体场景时，对照这张表直接选：

| **场景**                            | **推荐方案**           | **原因**                                |
| :---------------------------------- | :--------------------- | :-------------------------------------- |
| UI 开关（modal、dropdown、tooltip） | **useState**           | 局部、简单、独立                        |
| 单个或少量无关联的字段              | **useState**           | 独立更新，不需要协调                    |
| 多字段需要一起更新（局部）          | **object useState**    | 直接 spread，比 useReducer 简洁         |
| 状态逻辑复杂且需要单独写单测        | **useReducer**         | 纯函数 reducer 方便测试，其他场景不必用 |
| 稳定的全局配置（主题、语言）        | **Context + useState** | 变化少，不需要细粒度订阅                |
| 全局用户信息 / 权限                 | **Jotai atom**         | 多处订阅，动态更新                      |
| 跨页面共享的业务状态                | **Jotai atom**         | 跨路由持久，多处读写                    |
| 需要在组件外访问的状态              | **Jotai atom**         | atom 可在任意文件中 get/set             |
| 服务端数据 / API 缓存               | **React Query / SWR**  | 专门处理 loading/error/cache            |

# **7. 常见误用与纠正**

**❌ 误用 1：把只有一个组件用的状态放进 Jotai**

| **为什么错** atom 是为了跨组件共享而存在的。把组件内部状态放进 atom，会让它的生命周期脱离组件，还需要手动清理，徒增复杂度。 |
| :-------------------------------------------------------------------------------------------------------------------------- |

| <p>// ❌ 不必要：这个 modal 状态只有一个组件用</p><p>export const confirmModalOpenAtom = atom(false)</p><p></p><p>// ✅ 直接用 useState</p><p>const [isOpen, setIsOpen] = useState(false)</p> |
| :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

**❌ 误用 2：用 useReducer 解决「多个 useState 要同步」的问题**

| **为什么错** 多个 useState 同时更新，不是 useReducer 的专属场景，object useState 更简单直接。 |
| :-------------------------------------------------------------------------------------------- |

| <p>// ❌ 引入 useReducer 只是为了合并几个字段，得不偿失</p><p>dispatch({ type: 'FETCH_START' })</p><p></p><p>// ✅ object useState 已经够了</p><p>setFetchState(prev => ({ ...prev, isLoading: true, error: null }))</p><p></p><p>// ✅ 如果需要跨组件共享，用 object atom</p><p>setFetchAtom(prev => ({ ...prev, isLoading: true, error: null }))</p> |
| :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

**❌ 误用 3：用 Context 管理频繁变化的状态**

| **为什么错** Context 不做 diff，value 引用变化就全量重渲染。搜索关键词、筛选条件这类频繁变化的状态放 Context 里会有明显性能问题。 |
| :-------------------------------------------------------------------------------------------------------------------------------- |

| <p>// ❌ keyword 每次输入都变，Context 会让所有消费者重渲染</p><p>const SearchContext = createContext({ keyword: '', setKeyword: () => {} })</p><p></p><p>// ✅ 改用 Jotai，只有用到这个 atom 的组件才重渲染</p><p>export const searchKeywordAtom = atom('')</p> |
| :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

**❌ 误用 4：用手动 atom 管理服务端数据**

| **为什么错** 服务端数据天然有 loading / error / stale / cache 等复杂状态，手动用 atom 管理容易遗漏细节。React Query 或 SWR 专门解决这个问题。 |
| :-------------------------------------------------------------------------------------------------------------------------------------------- |

| <p>// ❌ 手动管理服务端状态，容易遗漏 cache/stale/refetch 等场景</p><p>const userDataAtom = atom<{ data: User | null; isLoading: boolean; error: string | null }>(...)</p><p></p><p>// ✅ 用 React Query，自动处理 loading/error/cache/refetch</p><p>const { data: user, isLoading, error } = useQuery({</p><p>`  `queryKey: ['user', userId],</p><p>`  `queryFn: () => fetchUser(userId),</p><p>})</p> |
| :------------------------------------------------------------------------------------------------------------ | --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

# **8. 项目约定**

- **📁 Atom 目录：**atom 文件统一放在 src/store/ 下，按功能模块命名，如 userAtom.ts、filterAtom.ts

- **📝 命名规范：**atom 变量以 Atom 结尾，如 userAtom、cartItemsAtom；派生 atom 也一样，如 isAdminAtom

- **⚡ hook 选择：**只读用 useAtomValue，只写用 useSetAtom，读写才用 useAtom。避免不必要的订阅

- **🔄 派生状态：**不要在多个组件里重复计算，统一用 computed atom 定义一次

- **🌐 服务端数据：**API 请求结果用 React Query / SWR 管理，不要用手动 atom 维护 loading/error

- **⬆️ 升级原则：**从最简单的方案开始（useState），感觉不够再升级，不要过度设计
